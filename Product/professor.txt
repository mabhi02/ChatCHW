Subject: Implementation of Cached, Multi-LLM Workflow for CHW Manual-to-DMN Conversion

Overview
To convert our Community Health Worker (CHW) manuals into Decision Model and Notation (DMN) files efficiently, we are moving away from stateless API calls. We will implement a Stateful "Maker-Checker" Pipeline using Context Caching. This reduces token costs by ~90%, ensures consistency across the ~24-prompt sequence, and utilizes a second "Red Team" LLM to validate clinical and logical accuracy.

1. Architecture: The Maker & The Red Team
We will utilize two distinct models to prevent "sycophancy" (where a model agrees with its own errors):

Maker (Gemini 1.5 Pro): Handles the long-term project state. Chosen for its 2M token window and persistent CachedContent (TTL-based storage).

Red Team (Claude 3.5 Sonnet): Acts as the validator. Chosen for its superior logical reasoning and "Prompt Caching" breakpoints.

2. The Caching Strategy
Instead of re-sending the manual with every prompt, we will use a Cumulative Freeze approach:

The Foundation Layer: Both models will have the full CHW Manual and the Master Variable Code List (ex_temp_c, etc.) in their cache.

The Progression: The Maker’s cache will be "refrozen" (updated) after each major artifact is finalized (e.g., Supply List → Variable Definitions → Decision Tables).

The Loop: 1. Maker generates an artifact.

2. Red Team compares the artifact against the cached Manual and DMN schema.

3. Repair: If the Red Team finds a bug, the Maker repairs it before the state is frozen and added to the "Gold Standard" history.

3. Implementation Details
Gemini: Use vertexai.generative_models.CachingValue. Set a TTL (Time to Live) of 24 hours for active development sessions.

Claude: Use the cache_control header in the messages API. Ensure the Manual is at the start of the block to maximize cache hits.

Red Team Analysis of the Above Proposal
Reviewer: Internal AI Auditor (Red Team Mode)

The "Infinite Growth" Flaw: The proposal suggests "refreezing" context after every prompt. If we append every raw chat turn to the cache, we will eventually hit "context saturation" where the model gets confused by old, rejected versions of artifacts.

The "Schema Blindness" Flaw: The Red Team is tasked with comparing the artifact to the manual, but DMN is an XML-based standard. If the Red Team doesn't have the DMN XML Schema (XSD) cached, it might approve a table that is clinically correct but technically "un-parseable" by our DMN engine.

The "Variable Drift" Flaw: The proposal assumes the "Master Variable List" is static. In reality, as we convert new chapters, the Maker will need to invent new codes. There is no mechanism described to sync these new codes back to the Red Team’s cache.

The "Cost Trap" Flaw: Gemini caches have a minimum charge (often 1 hour). If the script "refreezes" (creates a new cache object) after every prompt, we may inadvertently create 12–24 separate cached objects, spiking costs instead of lowering them.

Repaired Version (The "Production-Ready" Plan)
Refined Implementation Protocol
State Management (The "Gold Standard" Buffer): Do not cache the full conversation history. Only cache the current "Gold Standard" state (the Manual + the latest approved version of all artifacts). When an artifact is repaired, overwrite the old version in the next cache update to prevent the model from seeing its own past mistakes.

Schema Integration: Include the DMN 1.3/1.4 XML Schema definitions in the Foundation Layer for both the Maker and the Red Team to ensure all outputs are syntactically valid.

Variable Sync Loop:

Maker proposes a new variable code.

Red Team checks code against the "Naming Convention Logic."

Upon approval, the code is added to the Master Variable Registry which is updated in the next "Freeze" cycle.

Cache Management: To optimize costs, we will use one persistent Gemini Cache ID. Instead of creating a new cache per prompt, we will update the "Context" in batches of 3–4 prompts or at major chapter milestones.

Red Team Prompting: The Red Team will be explicitly prompted to "Look for what is MISSING," as models are historically better at seeing what is there than what has been omitted from the manual.

If you are using APIs, sending the full manual and prior DMN artifacts with every call is both slow and expensive.To minimize re-entering context, you should use Prompt Caching (also known as Context Caching). This allows the model to "remember" the static parts of your request (the manual, the DMN schema, and previous steps) so you only pay for and process the new prompt.

1. Provider-Specific Solutions
Each major AI provider handles context persistence differently via their APIs.

Google Gemini (Context Caching)
Gemini is particularly well-suited for your task because it supports very large caches (up to several million tokens) with a flexible "Time to Live" (TTL).


How it works: You create a CachedContent object containing your CHW manual and the DMN conversion rules.

Cost: You pay a small storage fee per hour, but you get a 75–90% discount on input tokens for every prompt that references that cache.


Best for: Iterative DMN generation where the source manual remains the same for hours of work.

Anthropic Claude (Prompt Caching)
Claude allows you to designate "breakpoints" in your prompt to be cached.


How it works: You use the cache_control block in the API. You can cache the manual and the first few generated DMN tables.

Cost: Significant savings (up to 90% cheaper) for "cache hits."


Constraint: Caches are generally shorter-lived (evicted after ~5 minutes of inactivity) compared to Gemini.

OpenAI (Automatic Caching)
OpenAI caches prompts automatically if they are over 1,024 tokens.


How it works: If the prefix of your prompt (the manual + previous artifacts) is identical to a previous call, OpenAI will automatically use the cache.


Constraint: You have less manual control; if you change even one word at the beginning of the prompt, the cache breaks.

2. Strategic Context Layering
To maximize cache hits, organize your API calls using a "layered" approach:

Layer	Content	Status
Layer 1: Static	The CHW Manual & DMN Specification	Always Cached
Layer 2: Growing	Verified DMN tables from previous steps	Updated & Re-cached
Layer 3: Dynamic	Current instruction (e.g., "Now convert Chapter 4")	New (Not Cached)
3. Implementation Workflow
Step A: The "Base" Cache
Create a cache that contains the "Source of Truth."

Content: The full CHW manual text + a "Style Guide" for how you want the DMN XML/JSON formatted.

Benefit: This is your largest block of tokens. Caching it once means you don't pay to "read" it for every subsequent table you generate.

Step B: The "State" Summary
Instead of sending every prior DMN artifact, send a Summary of the State.

If you are building a complex DMN with linked decisions, provide a list of existing Input Data and Decision names rather than the full XML of every previous table. This keeps the "Growing" layer small.

Step C: Use System Instructions
Place the "how-to convert to DMN" logic in the System Prompt rather than the User Prompt. System prompts are prioritized for caching in most architectures.

I put together a full architecture spec for both pieces you described. It's attached as ORCHESTRATOR.md. Here's the summary and the two things I need.

First, a note on scope: the chat interface I've been building (multi-model cross-validation, 6 parallel catchers, prompt caching, Haiku memory distillation, replay system) is overkill for what CHT needs to see. It's a powerful authoring environment but it's complex infrastructure that an open-source community wouldn't want to maintain. The attached spec is a fundamentally simpler approach that achieves the same goal with less moving parts.

The core idea: instead of the traffic cop checker pipeline with multi-agent red-team loops, we use a Recursive Language Model (RLM) approach. The key paper is "Recursive Language Models" by Zhang, Kraska, and Khattab at MIT CSAIL (January 2026, https://github.com/alexzhang13/rlm). I'd strongly recommend reading it. The short version: instead of feeding an entire manual into the model's context window (where quality degrades on longer documents), or relying on prompt caching (which also breaks down on longer manuals that exceed cache limits), we load the manual as a variable in a REPL (Read-Eval-Print Loop) programming environment and let the model programmatically examine, decompose, and extract from it through code execution and recursive sub-calls. The model never sees the full manual in its context window. It writes Python code to slice into sections, launch sub-calls per module, and build up the output incrementally. This scales to manuals of any length using the same architecture, and the model self-validates by calling deterministic checkers inside the REPL before returning its output. I attached the paper if you want to take a quick skim.

The input is the JSON guide (from the PDF extraction pipeline), and the final output is a validated JSON containing all clinical decision logic: Activator/Router tables, per-module decision tables (boolean-only), predicates with fail-safe defaults, and the full phrase bank. That JSON then gets deterministically converted to DMN XML, Mermaid, and XLSForm. The model never produces XML or XLSX directly.

Importantly, nothing in the system prompt or any prompt is specific to WHO 2012 or any particular manual. The prompts define the extraction architecture (Activator/Router pattern, boolean-only DMN, predicate conventions, variable naming, provenance requirements) but the actual clinical content: which danger signs exist, what thresholds apply, which modules the manual covers all comes from whatever JSON guide is loaded into the REPL. The same prompts should work on any CHW manual.

The whole thing runs through a basic web interface where the user pastes their own Anthropic API key to start a session --> no auth system, the API key is the auth. The team (and eventually CHT developers) can each use their own key, see the full REPL trajectory in real time, inspect every sub-call, and download the output artifacts. I've built this BYOK (bring your own key) session pattern before in another project and it works well for dev tooling where you want zero friction but full visibility.

The system prompt is the single most important artifact in this pipeline. It defines the extraction conventions and architecture but must remain fully manual-agnostic --> no hardcoded danger sign lists, no specific thresholds, no module names. The model discovers all of that from the guide itself.

2. VALIDATORS: ARE THESE SUFFICIENT?

The spec includes four validator modules that run inside the REPL (the model calls validate() and sees errors in real time). Here's what they check:

- Architecture: Activator exists (COLLECT policy), Router exists (FIRST policy), Router row 0 is the emergency short-circuit, every module in the Activator has a Router entry and a decision table, no module ID collisions, Integrative table exists
- Completeness: Every predicate referenced in DMN tables exists in the predicates array, every message ID in DMN outputs exists in the phrase bank, a composite danger sign predicate exists, all fields have provenance

- Naming: All variables follow the prefix convention (q_, ex_, v_, p_, dx_, tx_, ref_, m_, mod_, etc.)

These validators are also fully manual-agnostic. They check structural properties, not clinical content.
